# Post-install CLAUDE.md samples

Four examples of what a target repo's `CLAUDE.md` looks like **after**
installing ceo-orchestration and filling in the `{{PLACEHOLDER}}` values.
They are rendered instances of [`templates/CLAUDE.md`](../../templates/CLAUDE.md)
— copy the one closest to your stack, or use them to sanity-check your own
post-install edit.

All project names are fictional (`acme-*`).

## Index

| Sample | Stack | Shows |
|--------|-------|-------|
| [`node-api/CLAUDE.md`](node-api/CLAUDE.md) | Node.js 20 + TypeScript + Fastify | `--stack node` overlay (tsc + vitest pre-commit gate), strict-TS rules, route/service layering |
| [`django/CLAUDE.md`](django/CLAUDE.md) | Python 3.12 + Django 5 + DRF | Migrations discipline (never edit applied migrations, `makemigrations --check` in CI), ORM rules |
| [`go-service/CLAUDE.md`](go-service/CLAUDE.md) | Go 1.22 + chi + pgx | Table-driven tests, `%w` error wrapping, context propagation, `-race` gate |
| [`monorepo-ts/CLAUDE.md`](monorepo-ts/CLAUDE.md) | pnpm + Turborepo (Next.js + Fastify) | Affected-package test scoping (`--filter='...[origin/main]'`), package boundaries, backend + frontend teams |

## What every sample has in common

Each file keeps the structure the installer ships in
[`templates/CLAUDE.md`](../../templates/CLAUDE.md):

- **§0 Session Protocol** — Gates 1–3 pointing at the *installed* paths:
  `PROTOCOL.md` pointer at the target root, the `ceo-orchestration` skill at
  `.claude/skills/core/ceo-orchestration/SKILL.md`, `.claude/team.md`
  (plus `.claude/frontend-team.md` where the repo has a frontend).
- **Inline governance** — spawn protocol (`## AGENT PROFILE` /
  `## SKILL CONTENT` / `## FILE ASSIGNMENT`, enforced by
  `check_agent_spawn.py`), review + security VETOes, 3-strike, no auto-commit.
- **§1 Quick Reference** — the stack's real test / lint / run commands, so
  spawned agents verify with the same commands CI runs.
- **§3 Critical Rules** — stack-specific discipline plus the standard CEO
  rules: plan naming `PLAN-<NNN>-<slug>.md`, `/debate start` for L3+,
  worktree isolation for 3+-file refactors.

## How these repos were installed

```bash
# node-api
bash scripts/install.sh ~/projects/acme-api --stack node

# django / go-service (no stack overlay — see note below)
bash scripts/install.sh ~/projects/acme-crm
bash scripts/install.sh ~/projects/acme-gateway

# monorepo-ts (backend + frontend profiles)
bash scripts/install.sh ~/projects/acme-platform --stack node --profile core,frontend
```

**Honest note on `--stack`:** the installer currently ships one stack
overlay, `node` (adds a tsc + vitest pre-commit gate via
`templates/settings/settings.stack.node.json`). There is no `django` or
`go` overlay today — those samples install with the default `--stack none`
and wire their gates through their own CI, as shown in each Quick
Reference table. See `scripts/install.sh --help` for current options.

These samples are static documents; they were not generated by running the
installer against real repos. The `.claude/` paths they reference are the
ones `scripts/install.sh` copies into a target
(skills, hooks, commands, plan schemas, team files, plus the root
`CLAUDE.md` and the `PROTOCOL.md` pointer).

## After copying a sample

1. Replace every `acme-*` value with your project's real commands, paths,
   endpoints, and counts — a Quick Reference that lies is worse than none.
2. Delete the "Sample." banner at the top.
3. Keep the CHANGELOG stub and fill it at each session closeout.
