ceo-orchestration

Governance + agent orchestration protocol for Claude Code projects. Stdlib-only. Mechanically enforced. Evidence-grounded.

GitHub Quickstart Protocol Complete guide

What it is

A structured way to use Claude Code as if it were a multi-person engineering organization — with a CEO (Claude), VPs, specialists, staff with veto power, a spawn protocol, a plan→debate→execute workflow, and a skill library organized into core + frontend + optional domain profiles.

The framework ships mechanical governance: 12 Python hooks enforce invariants that can't be weakened by prompt wording. It's not a prompt pack. It's a runtime.

By the numbers

  • 48 skills (19 core + 8 frontend + 21 domain)
  • 12 Python hooks (stdlib only, Python 3.9+)
  • 41 ADRs published
  • 1,742 tests
  • 5 domain squads (fintech, edtech, government, lgpd-heavy-saas, trading-hft)
  • 16 SPEC v1 schema files
  • 0 runtime dependencies

Design decisions worth knowing

English is the Single Source of Truth (post v1.4.0)
PT-BR mirrors remain. Drift checker enforces structural parity across 7 document pairs (headings, code fences, line delta ≤10%, bidirectional cross-links).
Stdlib-only Python (ADR-002)
Hooks and scripts ship with zero pip installs. urllib over HTTP. No SDK vendor lock-in. Live adapter layer ships 4 providers (Claude / Gemini / OpenAI / local) as hand-rolled urllib wrappers behind a shared LiveCallPolicy.
24h RC hold, mechanically enforced
GA tags reject if the prior RC tag is less than 24 hours old (Codex re-pass window, ADR-103). No policy without mechanism (PLAN-013 Phase 0 item 0.3).
Plan → Debate → Execute
L3+ tasks require parallel agent critiques with ≥77% consensus threshold before execution. Plan artifacts + debate rounds + consensus docs are committed to .claude/plans/ as source of truth.
Canonical-edit hook
Direct edits to .claude/team.md, PROTOCOL.md, SKILL.md files require Owner-signed sentinels. Mechanical enforcement via check_canonical_edit.py (ADR-010 v1.1).
48 skills held steady
Framework deliberately does NOT expand skill count every sprint. Machinery sprints (multi-LLM adapters, retrieval, judge, marketplace) happen around the existing skill library.
Not public yet
Repo stays private until post-adopter validation in adopter-1 + adopter-2 (Sprint 15/16). Public launch is a Sprint 17 go/no-go decision with Owner approval, not an automatic milestone.

Install (target project)

cd /path/to/your-project
bash /path/to/ceo-orchestration/scripts/install.sh . --profile core,frontend

Domain profiles: --profile core,fintech or core,edtech,frontend, etc. See INSTALL.md.

How it works in one paragraph

You open Claude Code inside a project that has .claude/ installed. Claude reads CLAUDE.md, activates the ceo-orchestration skill, loads the team roster, and operates as the CEO. Every delegation happens through the Spawn Protocol (persona + skill + file assignment), enforced by hooks. Every L3+ decision runs through a debate with ≥2 parallel agents before execution. Every merge goes through a code-reviewer VETO. Everything is audited to ~/.claude/projects/<slug>/audit-log.jsonl. You read the audit log when something surprises you.

This is probably NOT for you if

  • You want a ChatGPT-style chat wrapper — use Claude Code directly
  • You want a prompt library — try karpathy-skills or similar curated collections
  • You want an npm install + auto-deploy — this framework installs via bash script and does not publish to npm by default (Sprint 17 conditional)
  • You want something lightweight — this is a runtime with 12 hooks + 48 skills + 1,742 tests. Run it because you want mechanical governance, not despite it.
