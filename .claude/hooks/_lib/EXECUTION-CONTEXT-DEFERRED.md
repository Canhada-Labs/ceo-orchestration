# execution_context.py — cross-process wiring DEFERRED (honest deferral)

> **Status:** DEFERRED (Owner-ratified, S154 — PLAN-112-FOLLOWUP-execution-context-wire, decision `RESCOPE-DEFER`).
> **Finding:** `F-1.2-execution_context-e33bc901` (PLAN-112 audit, eixo 1.2, P0).
> **Module:** `.claude/hooks/_lib/execution_context.py` (PLAN-102 Wave B.2b, shipped v1.36.0 / S142).
> **ADR:** ADR-133 (ACCEPTED) — coordinator-process-owned in-memory key + process-isolation invariant.

## What works today (intra-process only)

`ExecutionContext.sign()` -> `ExecutionContext.validate()` is a correct,
tamper-evident HMAC-SHA256 handshake **within a single Python process**:

- `sign()` returns hex HMAC over the canonical (`sort_keys`, no-space)
  payload using `_coordinator_key`.
- `validate()` rejects tampered payloads (`bad_signature`), stale contexts
  (`stale_nonce`), and replays (`nonce_replay`), all constant-time.

This intra-process value is real and is retained — it is exercised by
`tests/test_execution_context_deferral.py`.

## What does NOT work (the cross-process gap)

The original PLAN-102 goal — coordinator process A `sign()`s a spawn-handoff
payload, and a PreToolUse hook process B (`check_agent_spawn`) `validate()`s
it — is **cryptographically infeasible as built**:

1. **The key is in-memory and per-process.**
   `ExecutionContext._coordinator_key = secrets.token_bytes(32)` is a
   class-level value generated by `regenerate_key()` at coordinator process
   start, and is **NEVER persisted to disk** (module doctrine, lines 18-21).
2. **A hook is a fresh ephemeral process.** `check_agent_spawn` runs as a
   new Python interpreter on every PreToolUse. In that process
   `_coordinator_key is None`, so `validate()` short-circuits to
   `(False, "no_key")` at `execution_context.py:156-157` — **before any HMAC
   comparison**. Process A and process B do not share the key.
3. **The coordinator is a scaffold.** `.claude/scripts/swarm/coordinator.py`
   refuses to spawn loops (`"scaffold_only: worktree_orchestration_deferred"`)
   and never imports `execution_context`. There is no production dispatch
   site to wire `sign()` into.

## Explicit non-claim

This module does **NOT** provide a "replay-protected handoff" between the
coordinator and the spawn-validator. The monotonic nonce + 60s freshness
window are a **per-process** replay defense; they give **no cross-process
protection by construction** (a fresh process has an empty `_seen_nonces`
and a `None` key). Do not describe the spawn path as tamper-evident or
replay-protected on the basis of this module.

## Why we did NOT just "wire it"

Persisting or env-var-passing the key to "make the wire work" would:

- change the ADR-133 threat model (key-at-rest needs envelope-encryption +
  rotation; otherwise the tamper-evidence secret becomes an exposed secret),
- add hot-path file I/O + attack surface to every spawn for **zero** security
  gain (a leaked key signs forgeries as readily as it verifies),
- require ADR-133-AMEND-1 + a fresh L3 debate.

That is explicitly OUT of scope for this followup (Wave A debate S154:
identity-trust-architect + performance-engineer + qa-architect all
ADJUST_PROCEED, 0 VETO; Owner-ratified). Naively wiring `validate()` into the
hook would instead produce a **vacuous always-False gate** (`no_key` on every
call) that looks like security but enforces nothing —
`tests/test_execution_context_deferral.py` encodes this as a regression guard.

## Re-wire pre-conditions (both required)

The cross-process wire may be reopened only when BOTH land:

1. **Coordinator exits scaffold** — `.claude/scripts/swarm/coordinator.py`
   gains a real autonomous-loop dispatch site that imports `execution_context`
   and calls `sign()` on each handoff.
2. **A cross-process key design lands via ADR-133-AMEND-1** — e.g. a derived
   key (KDF from a sealed sentinel), an envelope-encrypted key-at-rest with
   rotation, or an IPC key-handoff — passing a fresh L3 debate on the amended
   threat model.

Until then: the two audit actions `execution_context_signed` and
`execution_context_validation_failed` remain **RESERVED** (registered in
`_KNOWN_ACTIONS` + SPEC v1, zero producers) so the allowlist/SPEC contract is
preserved for a future rebind without re-litigating the schema.
