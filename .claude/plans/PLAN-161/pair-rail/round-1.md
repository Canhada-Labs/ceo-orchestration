Findings:

1. **P1 — C5 corrupts the existing Codex-outage signal.** The plan emits `pair_rail_codex_unavailable` from the inverted Stop rail (`.claude/plans/PLAN-161-maintenance-sweep.md:280-290`), where Codex is the author and Claude is the reviewer (`.claude/hooks/check_codex_stop_review.py:6-13`, `.claude/hooks/check_codex_stop_review.py:61-67`). The audit schema defines that action specifically as Codex being unavailable (`SPEC/v1/audit-log.schema.md:315-316`, `SPEC/v1/audit-log.schema.md:880-901`), and the dispatcher counts every such event toward the Codex-outage predicate (`.claude/dispatcher/disable_predicate_eval.py:81-89`, `.claude/dispatcher/disable_predicate_eval.py:283-309`, `.claude/dispatcher/routing-matrix.yaml:281-288`). Because only healthy events are deduplicated, repeated Claude-review failures could falsely disable pair-rail routing. **Change:** introduce provider-neutral Stop-review health/fail-open actions, register their schema/scrubbing/classification cascade, and rate-limit or deduplicate failure events.

2. **P1 — The sentinel table incorrectly marks a kernel file as non-kernel.** The plan marks `.claude/hooks/_lib/audit_emit.py` “Kernel? no” (`.claude/plans/PLAN-161-maintenance-sweep.md:127-145`) and allocates kernel ceremony only for settings and workflow edits (`.claude/plans/PLAN-161-maintenance-sweep.md:299-304`). The kernel registry explicitly includes this file (`.claude/hooks/check_arbitration_kernel.py:76-91`). The consensus repeats the same two-hunk assumption (`.claude/plans/PLAN-161/debate/round-1/consensus.md:63-69`). **Change:** mark `audit_emit.py` as kernel-guarded and add a third kernel override segment with the required slug and acknowledgement.

3. **P1 — C1’s “complete five-surface atomic set” leaves normative assertions stale.** C1 declares five synchronized surfaces (`.claude/plans/PLAN-161-maintenance-sweep.md:214-225`), but ADR-158 normatively fixes `DENY_BASELINE` at seven entries (`.claude/adr/ADR-158-harness-config-gate.md:103-112`), and the permission design repeats the seven-rule baseline and the Edit/Write twins (`docs/PERMISSION-MODEL-DESIGN.md:290-301`). A second stale source anchor also exists in `.claude/hooks/check_harness_config.py:37-40`, beyond the anchor identified by the plan. If OQ5(a) is chosen, `docs/deny-baseline.md:95-104` also becomes false. The three listed ADR edits are in-place and do not themselves change the ADR count, but the atomic set is incomplete and requires at least a fourth in-place ADR amendment. **Change:** add ADR-158, the permission-model document, both harness anchors, and the conditional deny-baseline documentation to C1 and the sentinel scope.

4. **P1 — U1 misses a third target-tree writer and would break dry-run manifest loading.** The plan says exactly two writers require guards (`.claude/plans/PLAN-161-maintenance-sweep.md:226-231`). Besides `mkdir -p "$BAK_DIR"` at `scripts/upgrade.sh:567`, `_load_baseline_manifest` creates a temporary file inside that directory (`scripts/upgrade.sh:641-647`) and is called at `scripts/upgrade.sh:799`. Guarding only the directory creation makes this `mktemp` fail during dry-run, silently removing the baseline manifest needed for provenance-aware classification. **Change:** put the sanitized manifest temporary file in a secure location outside `$TARGET`, clean it via a trap, and add a semantic dry-run test proving manifest-based classifications and purge previews still occur—not merely a target-byte-identity test.

5. **P1 — C2 conflicts with existing tests that are omitted from every wave and scope table.** The proposed Grok artifact transport is specified at `.claude/plans/PLAN-161-maintenance-sweep.md:251-266`, but `scripts/tests/test-council-fixture.mjs:280-300` explicitly requires `redactor | vendor-cli`, while `.claude/scripts/tests/test_council_verify_semantics.py:177-197` requires one pipeline and rejects an unpiped redactor. Neither test file is in the sentinel scope, and W1’s selected pytest invocation only targets count tests (`.claude/plans/PLAN-161-maintenance-sweep.md:169-174`). **Change:** include both test files in W1 or W2, replace their one-pipe assertions with vendor-specific invariants, and run them in the staged oracle.

6. **P1 — C3’s literal `timeout` design is not portable to the repository’s macOS path.** The plan requires `timeout ${N}` (`.claude/plans/PLAN-161-maintenance-sweep.md:267-271`), while the repository’s installer already documents and handles platforms lacking GNU `timeout`, probing `timeout`, then `gtimeout`, then a bounded Python implementation (`scripts/install.sh:1099-1107`, `scripts/install.sh:1125-1135`). On the current macOS reviewer environment, neither `timeout` nor `gtimeout` is installed. **Change:** specify a portable process-group watchdog with the same fallback sequence and test the no-GNU-timeout branch. Also compute scope from actual bounded files/bytes rather than the brief string: the brief only contains the supplied scope text (`.claude/workflows/council-audit.js:126-140`), so `"."` is a one-character string representing the entire repository.

7. **P1 — C4’s acceptance criteria contradict its fail-fast design, and its timeout proof is incomplete.** The design intentionally fails when a pre-probe remains contended (`.claude/plans/PLAN-161-maintenance-sweep.md:272-279`), yet closeout says any load-window failure requiring a manual rerun is unacceptable (`.claude/plans/PLAN-161-maintenance-sweep.md:343-347`) and success requires surviving load without a rerun (`.claude/plans/PLAN-161-maintenance-sweep.md:427-429`). The cited estimate says `2×420 ≈940` (`.claude/plans/PLAN-161-maintenance-sweep.md:72-75`), although the capped attempts total 840 seconds; the additional backoff and probe budget is unspecified. The real workflow has a 16-minute job limit (`.github/workflows/validate.yml:1181-1183`), two 420-second attempts (`.github/workflows/validate.yml:1222-1231`), and separate control/floor steps (`.github/workflows/validate.yml:1272-1305`). The proposed 25-minute budget leaves only 240 seconds for every backoff, probe, smoke check, floor, and setup, while the floor’s 50 subprocesses have no hard timeout (`.claude/scripts/profile-opus-4-7.py:67-83`). **Change:** choose either bounded automatic recovery or an explicit infrastructure-failure acceptance model, then pin exact backoffs, thresholds, probe caps, setup headroom, and an auditable worst-case inequality.

8. **P1 — V1 will make its own staged oracle fail immediately because the newly governed documents are already stale.** The plan extends count enforcement to four documents (`.claude/plans/PLAN-161-maintenance-sweep.md:198-204`) but does not schedule their correction. Current false claims include `docs/ARCHITECTURE.md:47-51`, `docs/ARCHITECTURE.md:63-71`, `docs/FAQ.md:100-108`, `npm/README.md:44-60`, `npm/README.md:115-123`, and `docs/GUIA-COMPLETO.md:74-87`. The verifier’s own expected-count comments are also stale (`.claude/scripts/local/verify-counts.sh:29-40`). **Change:** derive current counts from disk, update all newly governed claims and verifier comments in V1, and seed tests with those corrected values before enabling tolerance-zero scanning.

9. **P2 — C2 does not mechanically guarantee mode `0600`, and its report schema has nowhere for the promised hash.** Redirecting the redactor into a temporary file (`.claude/plans/PLAN-161-maintenance-sweep.md:251-265`) creates a mode determined by the caller’s umask—commonly `0644`. Separately, the current lane schema has no artifact-hash field (`.claude/workflows/council-audit.js:109-118`). **Change:** set `umask 077` or exclusively pre-create/chmod the file before rename, verify its mode before invoking Grok, and either add an attestation field through schema/reduction/report output or remove the hash-reporting claim.

10. **P2 — The plan misattributes a “one-pipe” invariant to ADR-114.** ADR-114 requires redaction at every external call site but does not establish a universal single-pipeline transport invariant (`.claude/adr/ADR-114-structured-redactor.md:33-56`). **Change:** describe the Grok artifact-pointer rule as a new narrow transport exception while preserving ADR-114’s actual redaction-before-egress requirement and Codex’s stdin pipeline.

11. **P2 — L5 cites the opposite settings-upgrade branch.** The plan says upgrade “EXISTS-SKIPS settings.json” and cites lines 1235–1237 (`.claude/plans/PLAN-161-maintenance-sweep.md:336-341`). Those lines skip only when settings are missing; existing settings proceed through the additive merge (`scripts/upgrade.sh:1232-1248`). The conclusion that existing deny rules remain is plausible because the merge registers hooks rather than rewriting `permissions.deny`, but the stated mechanism is false. **Change:** cite the actual additive jq merge and say it leaves existing deny entries untouched; identify rules by exact strings rather than adopter-specific line numbers.

12. **P2 — A document already touched by C2 contains another false council claim.** `.claude/commands/council.md:52-59` says missing arguments silently default the audit scope to `.`, but the workflow rejects missing or empty scope (`.claude/workflows/council-audit.js:61-68`); only fixture mode receives a fallback (`.claude/workflows/council-audit.js:70-71`). **Change:** correct this while amending the council command documentation.
